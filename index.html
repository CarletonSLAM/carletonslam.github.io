About

The history of Carleton Stay Late And Make

  • Fall 2014

    Humble Beginnings

    Three senior students returned from their summer internships where they participated in weekly project development sessions. Wanting to bring these sessions to Carleton, they started Stay Late And Make - weekly meet-ups where students could get help for and work on whatever projects they wanted. Interest started growing and more students began attending .

  • Winter 2015

    Rise Above

    As the semester began, people were looking to tackle bigger projects. A small group of students worked on the WEATHER BALLOON COMPETITION and managed to secure funding for their project. This culminated in the launching of their weather balloon, which was retreived after rising for five hours through the atmosphere.

  • Summer 2015

    Quiet Times

    Although most people were away for the summer, some the SLAM members held a PCB building workshop that was a resounding success. Students designed their boards, printed them off, and soldered all the components on over the course of a few weekends.

  • Fall 2015

    Passing the Torch

    With the SLAM founders having graduated last semester, regulars from the previous sessions took over. More workshops were brought to the meetings, including Android app development, web development, and microcontroller programming. The future of SLAM looks bright, and it's just getting better and better.

SpaceGoldfish (Officially known as CU in the Air)

Project Members: Derek Stride, Irusha Vidanamadura, Krishna Mannem, Oscar Zambrano, Maurice Wier, Amente Bekele, Travis Ridge

CU in the Air is a group of Carleton University engineering students that have come together to put their love for creative thinking and problem solving to the test. As we are always looking for a challenge, the Global Space Balloon Challenge grabbed our attention immediately. Excited to see what kind of results we can produce, CU in the Air has officially joined the effort in celebrating an age where anyone can reach the edge of space for no more than a few hundred dollars.

The Tiva Launchpad (TM4C123G) was used as the microcontroller to measure Accelerometer, Barometer, and GPS location data, and send it through a Digital transmitter down to our base station. The payload also had a GoPro that was attached to a video transmitter. The base station consisted of a minivan with two antenna's on it: one for the digital transmitter and one for the analog transmitter. After the balloon was launched, the base station was used to catch up and stay in range with the balloon such that we received all the data.

Visit their Website, Github Repository or check out the page for the Global Space Balloon Challenge.

Proxima Education Robotics Platform (PERP)

Project Members: Hasith Vidanamdaura, Robert Nelson, and Darren Stahl

The Proxima Educational Robotics Platform is an ARM based robotics platform designed for teaching at Carleton.The objective of this project is to provide a consistent and modern hardware platform for use in courses throughout Carleton. This platform will be designed to provide a wide range of functionality, enabling multiple courses to base their lab work around it. In addition to the modernization of the course content, the platform will also help recruitment by providing a very tangible and visible product that Carleton can take to school fairs, and demonstrate to prospective students.

The robotics platform will have two primary hardware components, the wireless monitor, and the main peripheral board. Users interact with the main peripheral board by plugging in a primary microcontroller board that contains an ARM-based microcontroller running user-written programs. The pluggable interface allows flexibility in the selection of the primary microcontroller board, and the end user is not constrained to a single interface or a vendor for their needs. This strategy also maximizes the usable lifetime of the platform, as a vendor rendering a specific development board obsolete will no longer affect the usability of the platform.

Carleton Schedule Optimizer

Project by Sean Tohidi

This web application takes in Carleton University courses for a given semester. It grabs the courses' data off of Carleton's website and then finds a schedule made with those courses while minimizing the amount of breaks between classes. This project was written in Python, and the webpage was build using the Django framework.

October Hack The Month Challenge

Consume An API

Judjing Day: November 3rd

October's challange was to create a program that would consume an API. There are different kinds of APIs like REST APIs which use static web calls to return or alter data using GET and PUT requests respectively, and Graph APIs which are accessed in a similar way to that of REST but you can define specific fields that you would like to return. Some examples of different APIs available are the Twitter(REST) and Facebook(Graph) APIs. The goal of this challange is to consume the data from an API (Not nescesarilly these two) and do something interesting/creative/functional with it.

November Hack The Month Challenge

Build An API

Judjing Day: Decmber 1st

November's challenge is to create an API to be consumed by someone else. One way to build an API is to use a Rails/Django/Node.js application which can be hosted using a service such as Heroku to allow others to consume your API. In your web app you will create different routes that will become your different API calls. Using a web app framework you can also store data to your API using a database for things like user information or metrics if applicable. Another possible use of an API would be to return JSON containing the information needed for building a web page dynamically using javascript or any front end framework of your choice. For this challenge Documenting the different API calls will be important as well as documenting expected return values so that the APIs can be used by other people. This documentation should go in the README on the github project.

There is no front end required for this project we are simply looking at the API that you build. Judging will be based on how well the API works and how well it is documented. Having multiple commands is helpfull but not nescesary. Error handling is also helpful but unnescesary.
